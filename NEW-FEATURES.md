# What's new in the Klipsch India CRO demo vs the live store

This concept demo (Cinebels / Klipsch India) rebuilds the storefront with
conversion-focused enhancements that the live **klipschindia.com** does not have
today. Below are the five most impactful additions, why each one helps conversion,
and the live APIs the demo calls to stay grounded in real catalog data.

> Demo is a front-end concept; checkout is simulated (no real order/payment).

---

## The 5 new features

### 1. PDP Buying Guide (the headline feature)
A decision-support layer on every product detail page that the live store lacks entirely:
- A compact **"Quick Verdict + Best for"** strip directly below the price.
- A full **Buying Guide** below the buy box: *Specs at a glance*, *What the features
  mean* (a plain-language jargon buster), and *Good to know* (warranty / returns /
  payment / delivery).
- Generated by a **deterministic rules engine** from each product's own data — no
  external data, no runtime LLM.

**Why it converts:** audio gear is high-consideration. Explaining HDMI-ARC, Dolby
Atmos, Tractrix horns, and 24-bit decoding in plain language removes the research
detour that sends shoppers to Google (and to competitors), keeping the decision on-page.

### 2. Compare with search-to-add + type-ahead suggestions
Select up to 4 products and see a side-by-side modal (price, EMI, best-for, specs with
graceful blanks). On top of checkbox selection, the compare bar has a **search-to-add
box** with **type-ahead suggestion pills** (each with a "+", appearing after 3+ chars),
so a shopper can pull any product into the comparison without scrolling back to the grid.

**Why it converts:** comparison is the step right before purchase. Letting customers
build a comparison set instantly (without losing their place) shortens the path to a
confident choice.

### 3. Cart "You might also like" with lower-priced attach logic
The cart drawer shows a recommendations widget that deliberately surfaces
**lower-priced / accessory** items than what's already in the cart — never higher-priced
upsells.

**Why it converts:** cheaper add-ons have a far higher attach rate than pricier upsells.
Recommending down (a portable speaker next to a flagship, etc.) lifts units-per-order
and basket value without adding price friction at the decision moment.

### 4. Decision-first discovery: product finder + faceted shop
- A homepage **"Find your Klipsch"** finder that shops by use case (TV & movies, music,
  parties, travel, vinyl) and shortlists matching products.
- A **faceted shop page** (filter by use case, category, price; sort; deep-linkable
  `?cat=` / `?q=`).

**Why it converts:** most first-time buyers don't know model names. Letting them start
from intent ("I want big sound for parties") instead of a model list reduces bounce and
gets the right products in front of them faster.

### 5. "Klipsch New" experience toggle + the missing login & low-friction sign-in
- A header **"Klipsch New" toggle** switches the whole site to a CX/conversion-tuned
  redesign (denser, more scannable catalog grid; compacted hero so products are reachable
  sooner; stronger, higher-contrast CTAs; warmer, more welcoming palette; clearer price
  and trust hierarchy). The choice persists across pages and checkout.
- A **Sign-in button** (missing on the live site's header) plus a **one-tap login**
  (phone+OTP as the India default, Amazon sign-in, or true guest) — with auth deferred
  until order placement.

**Why it converts:** the toggle lets the team A/B the conversion-tuned layout against the
classic one with real customers. The added login + guest paths remove a common drop-off:
shoppers can build a cart with zero friction and only authenticate when they're ready to
buy. A persistent CLS fix (see below) also took the demo from a janky first paint to a
near-perfect layout-stability score.

## Live APIs the demo calls

Product data is **real**, captured from the Amazon **SmartBiz** storefront that powers
klipschindia.com (store id `62469`, host `api.smartbiz.in`). Because the SmartBiz API is
locked to the `klipschindia.com` origin (CORS), the data is captured from a browser
session on that origin and embedded into `assets/data.js` for the static demo.

| # | API / endpoint | What we read | Used for |
|---|----------------|--------------|----------|
| 1 | `GET api.smartbiz.in/stores/62469/v2/catalog/{sku}` | name, MRP, discounted price, discount %, description HTML, primary + secondary image URLs (gallery), colour, bestSeller flag, out-of-stock flag | The entire product catalog: pricing, the buying-guide specs/jargon engine, PDP galleries, stock state |
| 2 | SmartBiz Next.js `all-products` page payload (`featuredProductsList`) + recommendations campaign endpoint (`/campaigns/{id}/recommendations?sku=...`) | the set of live SKUs and which products are featured/recommended | SKU discovery — snowballing the full reachable product set (18 SKUs) |
| 3 | `attributes.buyingOptions` / `oos` flags (within the catalog response) | availability / in-stock state per variant | Stock badges and availability handling |

**Not a real API (clearly framed as simulated in the demo):** delivery **pincode
serviceability** — no public endpoint exists, so it's mocked. Ratings and offer payloads
were also kept static/illustrative for a hostable static site (the live offers endpoint
returned 401 to anonymous calls).

## Bonus: measurable quality, not just looks

Per the design intent, the redesign is built to convert, and that shows up in objective
metrics. Google **Lighthouse** (desktop) scores the demo **Performance 99 /
Accessibility 90 / Best Practices 100 / SEO 100** in both themes, versus roughly
**Performance 42 / Accessibility 73 / Best Practices 54 / SEO 91** for the live store.
Fixing a layout-shift issue (JS-injected header) alone moved the demo's CLS from ~1.0 to
~0.003 and Performance from 75 to 99. Full comparison in `WEBSITE-QUALITY-SCORECARD.md`.
